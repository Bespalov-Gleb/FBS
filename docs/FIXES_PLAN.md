# План исправлений: размеры и фото товаров

## 1. Размеры товаров

### Проблема
Неправильно подгружаются размеры — отображается не тот размер, который заказан (например, «Размер на модели: 50-52» вместо фактического размера заказа «46»).

### Причина
- **Ozon:** `get_product_sizes` возвращал атрибут «Размер» — только цифры (44). Нужен «Размер продавца» — формат 46-48 / M с буквами. Реальный размер заказа может быть в `dimensions.size_name` в продукте posting.
- **WB:** размер берётся из Content API по `chrt_id` — при несовпадении может подставляться первый размер из списка.

### Исправления (сделано)

1. **Ozon** (`order_sync_service.py`): при установке `p["size"]` сначала проверяем `dimensions.size_name` / `dimensions.size` из продукта posting. Только если их нет — используем `get_product_sizes`.

2. **Ozon** (`ozon.py`): в `get_product_sizes` приоритет атрибута **«Размер продавца»** (формат 46-48 / M) над «Размер» (только цифры). Ищем сначала «размер продавца», затем «размер».

3. **WB:** логика `_extract_size_from_card` уже использует `chrt_id` для выбора размера по конкретному варианту. Убедитесь, что `chrt_id` передаётся в metadata при синхронизации заказов WB.

### WB размер (дополнение)
- **`wildberries.py`**: в `_extract_size_from_card` — нормализация `chrt_id` (int/str), fallback на первый размер при несовпадении.

## 2. Фото товаров (второе вместо первого)

### Проблема
Во многих местах отображается второе фото (или штрихкод) вместо основного фото товара.

### Причина
- **WB Content API:** у части товаров первое фото в массиве `photos` — штрихкод или служебное изображение, а основное фото товара — второе.
- **WB CDN** (fallback без Content API): URL `1.webp` — первое изображение в CDN; у некоторых товаров оно тоже может быть штрихкодом.

### Исправления (сделано)

1. **WB Content API** (`order_sync_service.py`, `wildberries.py`): при наличии нескольких фото в `photos` берём **второе** (`photos[1]`) вместо первого. Если фото одно — используем его.

2. **WB CDN:** пока без изменений. При необходимости можно добавить `2.webp` в список URL при fallback (пробовать после `1.webp`).

### Ozon
Ozon использует `primary_image` из API — это корректное главное фото. Fallback на `images[0]` оставлен без изменений.

## 3. Резюме

| Проблема | Файл | Изменение |
|----------|------|-----------|
| Размер Ozon | `order_sync_service.py` | Приоритет `dimensions.size_name` из posting над `get_product_sizes` |
| Размер Ozon | `ozon.py` | Приоритет атрибута «Размер продавца» (46-48 / M) над «Размер» (только цифры) |
| Размер WB | `wildberries.py` | Нормализация `chrt_id`, fallback на первый размер |
| Фото WB | `order_sync_service.py`, `wildberries.py` | При нескольких фото — `photos[1]` вместо `photos[0]` |
| Ozon двойные заказы | `orders.py` | barcodes-pdf: по одному штрихкоду на каждый товар (многостраничный PDF) |

1. Синхронизировать заказы заново — чтобы подтянулись новые размеры и фото.
2. Проверить заказы Ozon и WB — корректность размеров и фото.
3. **Ozon размер из posting/fbs/get:** unfulfilled/list не возвращает dimensions. Добавлен вызов `posting/fbs/get` для каждого заказа — из products извлекаем dimensions.size_name или optional/required_product_attributes с «размер». Отключить: `FBS_OZON_FETCH_POSTING_DETAILS=0`.
